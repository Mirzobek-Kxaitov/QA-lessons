1-qism

Test Analysis

Asosiy atama

Test Analysis (Test tahlili) — talablarni o'qib, "NIMA test qilish kerak?" degan savolga javob topish bosqichi.

Oddiy qilib: Sizga berilgan vazifani chuqur tushunish va test uchun zarur narsalarni aniqlash.

Test Analysis bosqichlari

Yangi vazifa kelganda QA quyidagi 4 qadamni bajaradi:

  1. Talab/User Story'ni o'qish — diqqat bilan, har so'zga e'tibor
  2. Savollar tuzish — noaniq joylar, edge case'lar, missing scenarios
  3. Acceptance Criteria'ni tekshirish — yetarli'mi yoki qo'shimcha kerakmi?
  4. Test Conditions (test shartlari)ni aniqlash — nima testlanishi kerak?

Real misol: "Parolni tiklash" funksiyasi

PO bergan talab (asl)

"Foydalanuvchi parolni tiklay olishi kerak." — bu hammasi.

Sizning Test Analysis savollaringiz
  • Email orqali tiklash bo'ladimi yoki SMS orqali ham?
  • Havola necha vaqt ishlaydi (10 daqiqa, 1 soat, 1 kun)?
  • Bir marta ishlatiladimi yoki qayta-qayta?
  • Email mavjud bo'lmasa nima xabar chiqadi?
  • Necha urinishdan keyin akkount bloklanadi?
  • Eski parol va yangi parol bir xil bo'lishi mumkinmi?
  • Parol talablari (uzunlik, harf, raqam) bormi?
Sirli sir

Junior QA "Tushundim" deb darhol test case yozadi. Middle QA 10-15 ta savol beradi. Senior QA — bu savollarni 3 Amigos meeting'da beradi (oldindan).

2-qism

Test Design

Asosiy atama

Test Design (Test dizayni) — Analysis bosqichidan keyin "QANDAY test qilamiz?" degan savolga javob. Aniq test case'lar va ma'lumotlarni yaratish.

Oddiy qilib: Tahlil qilingan talab asosida amaliy test case'lar yozish.

Test Design ning 3 ta asosiy komponenti

📋
1. Test Conditions
TEST SHARTLARI
Test qilinishi kerak bo'lgan yuqori darajadagi shartlar. Bu hali aniq test case emas — "umuman nima testlanadi" ro'yxati.
Misol (parol tiklash):
• TC-1: Email orqali parolni tiklay olish
• TC-2: Noto'g'ri email kiritilganda xato
• TC-3: Havola muddati tekshiruvi
• TC-4: Yangi parol talablari
📝
2. Test Cases
TEST HOLATLARI
Har Test Condition uchun aniq, qadam-baqadam ko'rsatma. Test case to'liq strukturalanngan bo'lishi kerak.
Test Case formati:
• ID: TC-001
• Title: Email orqali parol tiklash
• Preconditions: Foydalanuvchi ro'yxatdan o'tgan
• Steps: 1. Login → 2. "Parolni unutdim" → 3. Email kiritish → 4. "Yuborish" bosish
• Expected: Email'ga havola yuboriladi
💾
3. Test Data
TEST MA'LUMOTLARI
Test bajarish uchun zarur aniq ma'lumotlar — email, parol, raqamlar va h.k. Bularsiz test bajarib bo'lmaydi.
Misol:
• Mavjud email: test@example.com
• Noto'g'ri email: noemail@example.com
• Yangi parol: NewPass123!
• Eski parol: OldPass456!

To'liq test case misoli (jadval ko'rinishida)

Test ID Tavsif Test Data Expected
TC-001 To'g'ri email bilan parol tiklash test@uzum.uz Email yuborildi
TC-002 Noto'g'ri email bilan urinish fake@uzum.uz "Email topilmadi"
TC-003 Bo'sh email bilan [empty] "Email kerak"
TC-004 Eski havolani ishlatish 1 soat oldingi havola "Havola eskirgan"
Muhim ketma-ketlik

Test Conditions → Test Cases → Test Data. Birinchi nima, keyin qanday, oxirida nima bilan. Bu ketma-ketlik buzilsa — sifatsiz test bo'ladi.

Test Design Texnikalari

Endi siz komponentlarni bilasiz. Lekin yana bir savol — qanday tanlash, qaysi test case'larni yozish kerak? 100+ test case yozish kerak emas, eng samarali bo'lganlarini tanlash kerak.

Buning uchun Test Design Texnikalari bor. 4 ta eng asosiysini ko'ramiz (kelgusi 13-darsda batafsil).

1
Equivalence Partitioning
EKVIVALENT BO'LIMLASHTIRISH
Ma'lumotlarni guruhlarga bo'lish va har guruhdan 1 ta qiymat bilan test qilish.
Misol: yosh maydoni (18-65)

3 guruh: juda yosh (<18), to'g'ri (18-65), juda katta (>65)

15 (yosh) 30 (to'g'ri) 70 (katta)

100 ta qiymat sinash o'rniga — 3 ta yetadi!

2
Boundary Value Analysis
CHEGARA QIYMATLAR TAHLILI
Chegara qiymatlarni alohida sinash. Eng ko'p bug aynan chegaralarda topiladi!
Misol: parol uzunligi (8-20 belgi)

Sinash kerak: 7, 8, 20, 21 (chegara va uning yon-atrofi)

7 belgi 8 belgi 20 belgi 21 belgi

Dasturchilar tez-tez > va >= ni adashtirib qo'yishadi — chegara testlari bunday bugni topadi.

3
Decision Table
QAROR JADVALI
Bir nechta shartlar birlashganda har xil natijalar bo'lganda ishlatiladi. Jadval ko'rinishida.
Misol: chegirma berish

Shartlar: Sotuv summasi > 1mln + Akkount eski (1+ yil)

State Transition
HOLAT O'TKAZISH
Tizim turli holatlardan o'tganda ishlatiladi (masalan, buyurtma holatlari).
Misol: Uzum buyurtma holatlari

Holatlar: Yangi → To'langan → Yetkazilmoqda → Yetkazildi

13-darsda batafsil

Bu 4 ta texnika — asoslari. Keyingi darsda ularning har birini batafsil amaliy misollar bilan ko'ramiz va Pairwise testing, Use Case testing kabi qo'shimchalarni ham o'rganamiz.

White Box texnikalar (qisqa eslatma)

Yuqoridagi 4 ta — Black Box texnikalari (10-darsda ko'rdik). White Box texnikalari ham bor, lekin ular asosan dasturchilar uchun:

  • Statement Coverage — har kod qatorini bir marta bajarish
  • Branch Coverage — har if/else shartni sinash
  • Path Coverage — barcha mumkin yo'llarni sinash

Junior QA bu atamalarni bilishi kifoya. Amalda foydalanmaysiz.

Q1 Test Analysis va Test Design farqi? +

Javob:

  • Test Analysis — "NIMA test qilamiz?" savoliga javob. Talablarni o'qib, savollar berish, Test Conditions aniqlash. Bu tahliliy ish.
  • Test Design — "QANDAY test qilamiz?" savoliga javob. Aniq Test Cases, Test Data yaratish, texnikalar qo'llash. Bu amaliy ish.

Misol:

  • Analysis: "Parol tiklash uchun nima testlash kerak?" → 8 ta Test Condition ro'yxati
  • Design: Har Test Condition uchun TC-001, TC-002... aniq qadamlar bilan

Pro javob: "Junior QA Design'da ishlaydi (tayyor case'larni bajaradi). Middle/Senior — Analysis va Design ikkalasini ham qiladi."

Q2 Test Case ning komponentlari qanday? +

Javob: "Standart Test Case 6 ta asosiy komponentdan iborat:"

  • Test ID — TC-001, TC-002 (unikal identifikator)
  • Title — qisqa sarlavha (masalan, "Login muvaffaqiyatli")
  • Preconditions — boshlanishidan oldin bajarilishi kerak (foydalanuvchi ro'yxatdan o'tgan)
  • Steps — qadam-baqadam ko'rsatma (1, 2, 3...)
  • Test Data — aniq ma'lumotlar (email, parol, raqamlar)
  • Expected Result — kutilgan natija

Qo'shimcha: Priority (P1/P2/P3), Severity, Author, Status — bu maydonlar Jira/TestRail'da avtomatik bo'ladi.

Q3 Equivalence Partitioning nima? +

Javob: "Equivalence Partitioning — ma'lumotlarni guruhlarga bo'lib, har guruhdan 1 ta qiymat bilan test qilish texnikasi."

Asosiy g'oya: Bir guruhdagi ma'lumotlar bir xil natija beradi, demak hammasini sinash shart emas.

Misol: Yosh maydoni 18-65 oralig'i

  • Invalid (yosh): 0-17 → bitta qiymat (masalan, 15)
  • Valid: 18-65 → bitta qiymat (masalan, 30)
  • Invalid (katta): 66+ → bitta qiymat (masalan, 70)

Natija: 100 ta test o'rniga 3 ta test yetadi.

Q4 Boundary Value Analysis nima va nima uchun muhim? +

Javob: "Boundary Value Analysis — chegara qiymatlarini alohida sinash texnikasi."

Nima uchun muhim? Statistika ko'rsatadi — buglarning 80% aynan chegaralarda topiladi! Dasturchilar tez-tez xato qilishadi:

  • > va >= ni adashtirish
  • < va <= ni almashtirish
  • Off-by-one xato (1 ta qiymat farq)

Misol: Parol 8-20 belgi

  • Sinash kerak: 7, 8, 20, 21 (chegara va yon-atrofi)
  • Equivalence + Boundary birga = eng samarali
Q5 Decision Table qachon ishlatiladi? +

Javob: "Decision Table bir nechta shartlar birlashganda ishlatiladi. Har shart kombinatsiyasi uchun aniq natija."

Qachon kerak?

  • Chegirma logikasi (bir nechta shartlar birga)
  • Foydalanuvchi ruxsatlari (admin/oddiy/mehmon)
  • To'lov qoidalari (valyuta + summa + foydalanuvchi turi)

Misol: Chegirma — VIP (ha/yo'q) + Summa >1mln (ha/yo'q) = 4 ta kombinatsiya

Afzallik: Murakkab logikani vizualizatsiya qiladi, hech bir holat o'tkazib yuborilmaydi.

Q6 Yaxshi Test Case ning xususiyatlari? +

Javob: "Yaxshi Test Case 5 ta asosiy xususiyatga ega:"

  • Aniq — har qadam ravshan, noaniq joy yo'q
  • Mustaqil — boshqa test case'ga bog'liq emas
  • Takrorlanuvchi — har gal bir xil natija beradi
  • O'lchanadigan — Pass/Fail aniq belgilanadi
  • Qisqa va aniq — uzun roman emas

Anti-pattern: "Loginni tekshirish" — bu yomon test case (juda umumiy).

To'g'ri: "TC-001: To'g'ri email va parol bilan login muvaffaqiyatli" — aniq.

Q7 Black Box va White Box texnikalar farqi? +

Javob:

  • Black Box texnikalar — kodni bilmasdan, kirish-chiqishga qarab. QA asosiy ishlaydi. Equivalence Partitioning, Boundary Value, Decision Table, State Transition.
  • White Box texnikalar — kodni ko'rib, har qator/shartni sinash. Dasturchilar ishlatadi. Statement Coverage, Branch Coverage, Path Coverage.

Misol:

  • Black Box: "Login formasiga to'g'ri va noto'g'ri qiymatlar kiritib sinash"
  • White Box: "Login funksiyasidagi har if/else shartni Unit testda sinash"

Junior uchun: "Men asosan Black Box texnikalarni ishlataman — Equivalence va Boundary Value har kundagi ishimda."
