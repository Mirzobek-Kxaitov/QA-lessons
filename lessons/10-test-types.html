Kirish

Test turlari nima uchun kerak?

Avvalgi darsda test darajalarini ko'rdik (Component, Integration, System, Acceptance). Endi boshqa savol — nimani test qilamiz?

Ko'pchilik o'ylaydi: "Testor faqat tugmalar ishlayaptimi tekshiradi". Aslida juda ko'p narsa test qilinadi — sayt qanchalik tez ochiladi, xavfsizmi, foydalanuvchi uchun qulaymi, turli brauzerlarda ishlaydimi.

Bu darsda siz test turlari tushunchasini o'rganasiz va intervyuda "Performance va Security testing farqi nima?" degan savolga aniq javob bera olasiz.

Katta rasm: 2 ta asosiy guruh

Barcha test turlari 2 ta katta guruhga bo'linadi:

⚙️

Functional Testing

"NIMA qiladi?" — sayt/ilovaning vazifasini tekshirish

🎯

Non-Functional Testing

"QANDAY qiladi?" — tezlik, xavfsizlik, qulaylik

Restoranga borasiz
Functional: "Buyurtma qilgan ovqat keldi" — vazifa bajarildi. · Non-Functional: "Ovqat 10 daqiqada keldi (tez)", "mazasi yoqdi (sifat)", "ofitsant yaxshi xizmat qildi (qulaylik)". Ikkinchisi ham juda muhim, lekin boshqa narsa.
1-qism

Functional Testing

Asosiy atama

Functional Testing (Funksional Testlash) — sayt yoki ilova o'z vazifasini to'g'ri bajarayaptimi tekshirish.

Oddiy qilib: "Tugma bosilganda mahsulot savatga qo'shiladimi?", "Login qilganda Dashboard ochiladimi?" degan savollarga javob.

Nima tekshiriladi?

  • Tugmalar — bosilganda nima bo'ladi?
  • Formalar — ma'lumot to'g'ri yuborildimi?
  • Linklar — to'g'ri sahifaga o'tadimi?
  • Hisoblashlar — savatdagi umumiy summa to'g'rimi?
  • Ma'lumotlar bazasi — ma'lumotlar saqlanayaptimi?
  • Biznes mantiq — xarid qilganda chegirma qo'llanadimi?
Uzum Market'da Functional test misollari
  • Login tugmasini bosaman → Dashboard ochiladi ✅
  • Mahsulot qidiruv maydoniga "telefon" yozaman → telefon mahsulotlari chiqadi ✅
  • "Savatga qo'shish" tugmasini bosaman → savat badge'i +1 ga o'zgaradi ✅
  • 2 ta mahsulotni savatga qo'shaman → umumiy summa to'g'ri hisoblanadi ✅
  • Noto'g'ri parol bilan kiraman → xato xabari chiqadi ✅

Kim va qanday bajaradi?

  • Kim: QA (sizning asosiy ishingiz!) — 80% vaqtingiz shunga ketadi
  • Qachon: System Testing darajasida
  • Qanday: Manual yoki Automation (Selenium, Playwright)
Asosiy nuqta

Functional testing — bu testorning kundalik ishi. Siz test case yozganda, ssenariy bajarayotganda, bug topganda — bu odatda Functional testing.

2-qism

Non-Functional Testing

Asosiy atama

Non-Functional Testing (Nofunksional Testlash) — sayt o'z vazifasini QANCHALIK yaxshi bajarayaptimi tekshirish.

Oddiy qilib: "Sayt tezmi?", "Xavfsizmi?", "Qulaymi?", "Turli qurilmalarda ishlaydimi?" kabi savollar.

Non-Functional testing bir nechta turga bo'linadi. 4 ta asosiysini ko'rib chiqamiz.

⚡
Performance Testing
UNUMDORLIK / TEZLIK
Saytning qanchalik tez va samarali ishlayotganligini tekshirish.
Sodda savol: Sayt 2 soniyada ochiladimi? Yoki 10 soniya kutasizmi?

Performance Testing 3 turga bo'linadi:

🏋️ Load Testing
Odatdagi yuk ostida sayt qanday ishlaydi?
Misol: 1000 ta foydalanuvchi bir vaqtda — sayt sekin bo'ladimi?
💥 Stress Testing
Qachon sayt "yiqiladi"? Chegarasini topish.
Misol: 10 000 ta foydalanuvchi — sayt o'chadimi yoki chidaydi?
📦 Volume Testing
Katta ma'lumot hajmi bilan ishlash.
Misol: 1 million mahsulot bor — qidiruv tezmi?
Kim bajaradi
Middle/Senior QA, Performance Engineer
Asboblar
JMeter, LoadRunner, k6
🔐
Security Testing
XAVFSIZLIK TESTLASH
Saytning xavfsizligini tekshirish — ma'lumotlar o'g'irlanadimi, hakerlar kira oladimi.
Sodda savol: Hakerlar mening karta ma'lumotlarimni ola oladimi? Mening parolim xavfsiz saqlanayaptimi?

Nima tekshiriladi?

  • SQL Injection — login formalarga maxsus kod yuborish bilan ma'lumotlar bazasiga kirish
  • XSS (Cross-Site Scripting) — saytga yomon JavaScript kod joylash
  • Authentication — parol talablari kuchlimi?
  • Authorization — oddiy foydalanuvchi admin sahifasiga kira oladimi?
  • Data Encryption — ma'lumotlar shifrlanganmi?
Kim bajaradi
Security Specialist, Senior QA
Asboblar
OWASP ZAP, Burp Suite
Sizning TestShop'ingizda: Login formaga SQL injection ' OR 1=1-- yozib ko'ring — agar kirgiza olsa, bu xavfsizlik bugi!
🎨
Usability Testing
QULAYLIK TESTLASH
Sayt foydalanuvchi uchun qanchalik qulay va tushunarli?
Sodda savol: Oddiy odam 1 daqiqada savatga mahsulot qo'sha oladimi? Yoki 10 daqiqa qidirishi kerakmi?

Nima tekshiriladi?

  • Intuitivlik — ko'rsatma o'qimasdan foydalanish mumkinmi?
  • Navigatsiya — menyular aniq va oddiymi?
  • Tezkor tushunish — birinchi 5 soniyada foydalanuvchi nima qilishini tushunadimi?
  • Xato xabarlari — aniq va yordamchi'mi?
  • Mobil qulaylik — telefonda ham oson'mi?
Kim bajaradi
QA, UX Designer, Haqiqiy foydalanuvchilar
Usul
Foydalanuvchi kuzatuv, A/B testing
🌐
Compatibility Testing
MOSLIK TESTLASHI
Sayt turli muhitlarda ishlaydimi? — brauzerlar, qurilmalar, OS.
Sodda savol: Sayt Chrome'da ishlaydi. Safari'da ham ishlaydimi? iPhone'da qanday ko'rinadi?

Nima tekshiriladi?

  • Brauzerlar: Chrome, Safari, Firefox, Edge
  • Qurilmalar: Desktop, telefon, planshet
  • Operatsion tizimlar: Windows, Mac, Android, iOS
  • Ekran o'lchamlari: Kichik, o'rta, katta ekranlar
  • Internet tezligi: 4G, 3G, Wi-Fi
Kim bajaradi
QA (manual + automation)
Asboblar
BrowserStack, LambdaTest
Real muammo: Sayt Chrome'da chiroyli, lekin iPhone Safari'da tugmalar joyidan chiqib ketgan — bu Compatibility bug!

3-qism

Box Testing — 3 usul

Test turlari uchun yana bir klassifikatsiya bor — testor kodni qanchalik biladi? degan savolga qarab.

💡 Oddiy misol
Avtomobilni tekshirish
Black Box: Haydovchisiz — haydab ko'rasiz, motor ishlayaptimi, tormoz yaxshimi? Ichkarisini bilmaysiz. · White Box: Mexanik — kapotni ochadi, har bir detalni ko'radi. · Gray Box: Sizga umumiy texnik hujjat berilgan — asosiy qismlarni bilasiz, lekin har bir gayka emas.
⬛
Black Box Testing
QA NING ASOSIY USULI
Testor dasturning ichki kodini bilmaydi. Faqat "kirish va chiqish"ga qaraydi.
Sodda misol: Login formaga email va parol kiritaman — Dashboard ochilishi kerak. Kod qanday yozilganini bilmayman, lekin natijani ko'raman.

Xususiyatlari:

  • ✅ Dasturchilik bilimi shart emas
  • ✅ Haqiqiy foydalanuvchi nuqtai nazari
  • ✅ System Testing darajasida keng qo'llaniladi
  • ❌ Kodning ichki mantig'ini sinab bilmaysiz
Kim bajaradi
QA (Junior/Middle/Senior)
Daraja
System, Acceptance
⬜
White Box Testing
DASTURCHI USULI
Testor butun kodni ko'radi va tushunadi. Har bir qatorni test qiladi.
Sodda misol: Dasturchi qo'shish funksiyasini yozdi. U kod ichiga kirib, har bir if va else shartini alohida sinab ko'radi.

Xususiyatlari:

  • ✅ Chuqur analiz — har bir kod yo'lini sinab bilasiz
  • ✅ Bug'larni ildizidan topish
  • ✅ Unit va Component testing uchun asosiy
  • ❌ Dasturlashni bilish shart
  • ❌ Juda ko'p vaqt oladi
Kim bajaradi
Dasturchi, Senior QA
Daraja
Unit, Component
◐
Gray Box Testing
ORALIQ USUL
Testor kodni qisman biladi — asosiy arxitekturani tushunadi, lekin har bir qatorni emas.
Sodda misol: Testor ma'lumotlar bazasi strukturasini biladi va SQL so'rovlarini o'zi yozib, ma'lumotlar to'g'ri saqlanayaptimi tekshiradi.

Xususiyatlari:

  • ✅ Black Box + White Box afzalliklari birga
  • ✅ Integration Testing uchun ideal
  • ✅ API testing'da ko'p ishlatiladi
  • ✅ Middle/Senior QA'lar uchun standart
Kim bajaradi
Middle/Senior QA
Daraja
Integration, API
Siz qaysi usulni ishlatasiz?

Junior QA asosan Black Box. Middle QA Black + Gray Box (API testing). Senior QA esa uchalasini ham ishlatishi mumkin. Dasturchilar esa White Boxda ustalar.

Intervyuga tayyorlov

Intervyu savollari

Test turlari — intervyu'da eng ko'p so'raladigan mavzulardan. 8 ta muhim savolga professional javoblarni o'rganing.

💡 Har savol ustiga bosing — professional javobni ko'rasiz.

Q1 Functional va Non-Functional testing farqi? +

Javob:

  • Functional Testing — "NIMA qiladi?" savoliga javob. Dasturning asosiy vazifalarini tekshiradi (tugma, forma, hisoblash).
  • Non-Functional Testing — "QANCHALIK yaxshi qiladi?" savoliga javob. Tezlik, xavfsizlik, qulaylik, moslik.

Misol: "Login ishlayaptimi?" — Functional. "Login 1 soniyada ochiladimi?" — Non-Functional (Performance).

Pro javob: "Men asosan Functional testing bilan shug'ullanaman (80% vaqt), lekin Security va Usability test'larda ham qatnashaman."

Q2 Performance, Load va Stress testing farqi? +

Javob: "Performance Testing — umumiy atama. Uning ichida 3 turi bor:"

  • Load Testing — odatdagi yuk ostida test. Misol: 1000 ta foydalanuvchi bir vaqtda.
  • Stress Testing — chegaraga yetkazib test. Misol: 10 000 ta foydalanuvchi — qachon yiqiladi?
  • Volume Testing — katta ma'lumot hajmi bilan test. Misol: 1 million mahsulot ustidan qidiruv.

Pro javob: "Black Friday kabi katta yuk oldidan Load Testing qilamiz — real yuk bilan sinab ko'ramiz. Stress bilan esa chegaralarni topamiz."

Q3 Black Box va White Box testing farqi? +

Javob:

  • Black Box — testor kodni bilmaydi. Faqat kirish-chiqishga qaraydi. Foydalanuvchi nuqtai nazari.
  • White Box — testor butun kodni ko'radi. Har qatorni sinab chiqadi. Dasturchi usuli.

Qachon qaysi biri?

  • Black Box — System va Acceptance testing. QA'larning asosiy usuli.
  • White Box — Unit va Component testing. Dasturchilar qiladi.
  • Gray Box — Integration/API testing. Middle QA'lar uchun.

Muhim: "Men Junior QA sifatida asosan Black Box ishlayman, chunki dasturchilik bilimim chuqur emas."

Q4 Security testing qanday qilinadi? +

Javob: "Security testing — saytning xavfsizligini tekshirish. Asosiy 4 yo'nalish:"

  • SQL Injection — login formalarga maxsus kod kiritish (' OR 1=1--). Ma'lumotlar bazasiga ruxsatsiz kirishga urinish.
  • XSS (Cross-Site Scripting) — saytga yomon JavaScript kod joylashtirish.
  • Authentication — parol talablari, 2FA bor-yo'qligi.
  • Authorization — oddiy foydalanuvchi admin sahifaga kira oladimi?

Asboblar: OWASP ZAP, Burp Suite. Lekin bu asosan Security Specialistning ishi. Junior QA faqat asosiy tekshiruvlarni qiladi.

Q5 Usability testing qanday o'tkaziladi? +

Javob: "Usability testing — sayt qanchalik qulay ekanligini tekshirish."

Usullar:

  • Haqiqiy foydalanuvchi kuzatuv — oddiy odamga vazifa beramiz, u qanday bajarishini kuzatamiz.
  • A/B Testing — ikki variantni solishtirish. Qaysi yaxshiroq?
  • Heuristic Evaluation — UX mutaxassislari saytni qoidalar bo'yicha baholaydi.

Nima tekshiriladi:

  • Birinchi 5 soniyada nima qilishini tushunadimi?
  • Xato xabarlari aniqmi?
  • Mobil versiyada oson'mi?
Q6 Compatibility testing'da nima tekshirasiz? +

Javob: "Compatibility testing — sayt turli muhitlarda ishlayaptimi tekshirish."

5 ta asosiy yo'nalish:

  • Brauzerlar: Chrome, Safari, Firefox, Edge
  • Qurilmalar: Desktop, telefon, planshet
  • OS: Windows, Mac, iOS, Android
  • Ekran o'lchamlari: 320px, 768px, 1920px
  • Internet tezligi: Wi-Fi, 4G, 3G

Asboblar: BrowserStack, LambdaTest — cloud orqali 100+ brauzerda test qila olasiz.

Pro javob: "Real misol — Chrome'da chiroyli ishlaydi, lekin Safari'da tugmalar joyidan chiqib ketgan. Bu Compatibility bug."

Q7 Siz qaysi test turlarida tajribangiz bor? +

Javob shabloni (halol): "Men Junior QA sifatida asosan quyidagi test turlarida ishlaganman:"

  • ✅ Functional Testing — kundalik ishim
  • ✅ Compatibility Testing — Chrome, Safari, mobil versiya
  • ✅ Usability Testing — sodda tekshiruvlar
  • ⚠️ Security Testing — asosiy darajada (SQL injection, XSS)
  • ❌ Performance Testing — hali o'rganayapman, JMeter ishlataman

Muhim: Halol javob bering. "Men hamma narsani bilaman" — yolg'on deb tushuniladi. Aniq nimani bilishingizni va o'rganayotganingizni ayting.

Q8 Katta e-commerce saytida qanday test turlari muhim? +

Javob: "Uzum, Click kabi katta e-commerce saytlarida 5 ta test turi muhim:"

  • 1. Functional — savat, to'lov, buyurtma — asosiy biznes oqimi to'g'ri ishlashi kerak
  • 2. Performance — Black Friday, aksiyalar paytida sayt yiqilmasligi kerak
  • 3. Security — karta ma'lumotlari, foydalanuvchi ma'lumotlari himoyalangan bo'lishi kerak
  • 4. Usability — foydalanuvchi 3 qadamda xaridni yakunlay olishi kerak
  • 5. Compatibility — mobil versiya ideal ishlashi kerak (50%+ trafik mobil'dan)

Muhim xulosa: "Katta e-commerce sayt uchun to'liq test strategiya kerak — faqat Functional yetarli emas. Har bir test turini inobatga olish muhim."
