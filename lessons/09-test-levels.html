Test darajalari va Regression

Test darajalari — qaysi bosqichda kim, nima test qiladi. Test piramidasi va Regression Testing — professional QA uchun asosiy tushunchalar.

Darajalar va strategiya

Kirish

Nega test darajalari kerak?

Tasavvur qiling: siz 100 qavatli bino quryapsiz. Har qavatda bir xil tarzda tekshirasizmi? Albatta yo'q. Poydevorni — muhandislar, xonalarni — sozlovchilar, umumiy ko'rinishni — arxitektor tekshiradi.

Dasturlashda ham shunday. Kichik qismlarni dasturchilar, modullar orasidagi bog'liqlikni senior dasturchilar, butun tizimni QA'lar, yakuniy qabul qilishni biznes tomoni sinaydi.

Bu darsda 4 ta test darajasini, ular orasidagi balansni (Test Piramidasi) va Regression Testing'ni ko'rib chiqamiz.

1-qism

2-qism

Test Piramidasi

Asosiy atama

Test Piramidasi — test darajalari orasida qancha test bo'lishi kerakligi haqidagi nazariya. Mike Cohn tomonidan taklif qilingan.

Asosiy g'oya: Ko'p sonli kichik (tez) testlar pastda, kam sonli katta (sekin) testlar yuqorida.

Piramidaning ideal ko'rinishi

Yuqorida UI / E2E Tests ~10%
3-daraja System Tests ~15%
2-daraja Integration Tests ~25%
1-daraja (eng pasti) Unit / Component Tests ~50%
↑ Tez, arzon, ko'p miqdor = pastda · Sekin, qimmat, kam miqdor = yuqorida ↓

Nega bunday proporsiya?

Pastdagi testlar (Unit)

Afzalliklari: Juda tez bajariladi (millisekunddalarda), arzon yozish, ishonchli. Shuning uchun ko'p miqdor yozish mumkin va kerak.

Yuqoridagi testlar (E2E)

Kamchiliklari: Sekin (minutlarda), qimmat, ba'zida "flaky" (bir xil natija bermaydi). Shuning uchun kam, faqat eng muhim sahnalar uchun.

Amaliy xulosa

Agar sizning loyihangizda 1000 ta Unit test va 10 ta E2E test bo'lsa — bu sog'lom piramida. Agar 100 ta Unit va 500 ta E2E — bu muammo (Ice Cream Cone).

Anti-pattern: Ice Cream Cone (muzqaymoq konus)

Ba'zi loyihalarda piramida teskari bo'ladi — ko'p E2E testlar, kam Unit testlar. Bu yomon holat:

⚠️ Ice Cream Cone Anti-Pattern
Manual Testing ~40%
E2E / UI Tests ~30%
Integration ~20%
Unit ~10%
Bu holat sekin, qimmat va nazorat qilish qiyin!
💡 Hayotiy misol
Poydevor va tom
Bino qurayotganingizda kuchli poydevor va yengil tom kerak. Agar poydevor zaif, tom og'ir bo'lsa — bino yiqiladi. Xuddi shunday testlarda ham — ko'p Unit test va kam E2E test — bu kuchli strategiya.
3-qism

Regression Testing

Asosiy atama

Regression Testing — dasturga yangi o'zgarish kiritilgandan so'ng, eski funksiyalar ishlayaptimini tekshirish.

Oddiy qilib: yangi feature qo'shildi — qolgan narsalar yana ishlayaptimi? Regression Testing aynan shuni tekshiradi.

Nima uchun kerak?

Real misol

Uzum Market'da yangi "Chegirma kodi" funksiyasi qo'shildi. Dasturchi kodni yozdi, yangi funksiya ishlayapti. Lekin... endi savat ham ishlamay qoldi! Chunki yangi kod eski savat kodiga ta'sir qilgan. Bu holatni oldini olish uchun — Regression Testing.

Qachon bajariladi?

  • Yangi funksiya qo'shilgandan so'ng — boshqa narsalar buzilmaganligini tekshirish
  • Bug tuzatilgandan so'ng — tuzatish boshqa joyni buzmaganmi?
  • Har release'dan oldin — production'ga chiqishdan oldin
  • Konfiguratsiya o'zgarganda — server, database sozlamalari o'zgarganda

Manual vs Automation

Regression Testing juda ko'p marta takrorlanadigan ish. Shuning uchun u avtomatlashtirish uchun ideal. Keling qaraymiz:

Manual Regression

Yangi jamoalarda yoki kichik loyihalarda ishlatiladi.

  • Asosiy senariylar qo'lda tekshiriladi
  • Har release oldidan 1-2 kun ketadi
  • Inson xatosi bo'lishi mumkin
  • Yangi boshlovchilar uchun yaxshi

Automation Regression

Katta loyihalarda albatta kerak.

  • Yuzlab testlar 30 daqiqada
  • Har night'da avtomatik ishga tushadi
  • Bug'larni darhol topadi
  • Senior QA yoki SDET yozadi

Smoke Testing vs Regression Testing

Bu ikkala atamani ko'pchilik aralashtirib yuboradi. Intervyu'da so'raydi.

Smoke Testing

Smoke — "ishlayaptimi umuman?" degan savolga javob. Eng asosiy funksiyalar ishlayaptimi? 15-30 daqiqalik tez tekshiruv.

Misol: Sayt ochilayaptimi? Login ishlayaptimi? Bosh sahifa chiqayaptimi? Agar ha — keyingi testlarga o'tamiz.

Regression Testing

Regression — "eski narsalar yana ishlayaptimi?" degan savolga javob. Chuqurroq, batafsilroq tekshiruv. Bir necha soatdan bir necha kungacha.

Misol: Barcha eski funksiyalarni bir-bir tekshirish. Login, ro'yxatdan o'tish, savat, to'lov, profil... hammasini.

Oddiy farqlash

Smoke = tez, yuzaki ("ishlayaptimi?") · Regression = sekin, chuqur ("to'liq ishlayaptimi?")

Intervyu savollari

💡 Har savol ustiga bosing — professional javobni ko'rasiz.

Q1 Test darajalarini sanab bering va kim bajarishini ayting. +

Javob: "4 ta asosiy test darajasi bor:"

  • Component/Unit Testing — Dasturchilar bajaradi. Eng kichik qism — bir funksiya.
  • Integration Testing — Dasturchi + QA. Modullar o'zaro ishlayaptimi.
  • System Testing — QA'lar bajaradi. Butun tizim end-to-end.
  • Acceptance Testing — Product Owner yoki mijoz. Biznes talablar bajarilganmi.

Muhim: "Men aynan System Testingda ishlayman — bu QA'ning asosiy sohasi."

Q2 Test Piramidasi nima va u nima uchun muhim? +

Javob: "Test Piramidasi — test darajalari orasida balans haqidagi nazariya. Ko'p Unit, o'rtacha Integration, kam E2E."

Nima uchun?

  • Unit testlar tez (millisekund) va arzon
  • E2E testlar sekin (minut) va qimmat
  • Ko'p arzon test = tez feedback = pul tejash

Anti-pattern: Ice Cream Cone — ko'p E2E, kam Unit. Bu yomon — tez natija yo'q.

Q3 Regression Testing nima? Qachon bajariladi? +

Javob: "Regression Testing — dasturga yangi o'zgarish kiritilgandan so'ng, eski funksiyalar buzilmaganligini tekshirish."

Qachon bajariladi:

  • Yangi funksiya qo'shilganda
  • Bug tuzatilgandan so'ng
  • Har release'dan oldin
  • Konfiguratsiya o'zgarganda

Pro javob: "Katta loyihalarda Regression Testing avtomatlashtirilishi kerak — chunki u har safar takrorlanadi. Manual qilish ko'p vaqt oladi."

Q4 Smoke Testing va Regression Testing farqi? +

Javob:

  • Smoke Testing — "ishlayaptimi umuman?". Eng asosiy funksiyalar (login, bosh sahifa). 15-30 daqiqa.
  • Regression Testing — "barcha eski funksiyalar ishlayaptimi?". Chuqur, batafsil. Soatlar yoki kunlar.

Ketma-ketlik: "Build kelsa, avval Smoke qilamiz. Agar smoke o'tsa — keyin Regressionga o'tamiz. Smoke Fail bo'lsa — Regression'ga o'tmaymiz, build'ni qaytaramiz."

Q5 Unit Testing bilan Integration Testing farqi? +

Javob:

  • Unit Testing — bitta komponent (masalan, bir funksiya) alohida test qilinadi. Boshqalardan ajratilgan.
  • Integration Testing — 2 yoki undan ko'p komponent birga ishlayaptimi tekshiriladi.

Misol:

  • Unit: add(2, 3) = 5 — bitta funksiya
  • Integration: Login formasidan yuborilgan ma'lumot → server → database'ga yetib borayaptimi — 3 ta komponent birga
Q6 Siz qaysi test darajasida ko'proq ishlaysiz? +

Javob shabloni: "Men asosan System Testingda ishlayman — bu QA'ning asosiy sohasi. Butun tizimni end-to-end sinab ko'raman."

Qo'shimcha:

  • Integration Testingda ham qatnashaman — API testing uchun (Postman)
  • Acceptance Testingda PO bilan birga ishlayman
  • Unit Testing bilan to'g'ridan-to'g'ri ishlamayman — bu dasturchilarning sohasi

Pro javob: "Lekin Unit test natijalarini ko'rib boraman — agar unit test'larda muammo bo'lsa, bu kelgusida System darajasida ham chiqadi."

Q7 Acceptance Testing'da QA qanday vazifa bajaradi? +

Javob: "Acceptance Testing'ni asosan Product Owner yoki mijoz bajaradi. Lekin QA:"

  • UAT muhitini tayyorlashda yordam beradi
  • Test ma'lumotlarini tayyorlaydi
  • PO'ga test senariylar tushuntiradi
  • UAT paytida topilgan bug'larni qabul qilib, developer'larga yuboradi
  • Tuzatilgan bug'larni qayta tekshiradi

Muhim: "UAT — QA'ning qaror qabul qilish sohasi emas. QA biznes emas, texnik mutaxassis."
