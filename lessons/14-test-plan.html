Test Plan nima?

Tasavvur qiling: PO sizga keldi va dedi: "Yangi loyiha boshlanmoqda. Sen QA Lead'sen. 2 oydan keyin chiqaramiz." Siz nima qilasiz?

Junior QA: "Test case yozaman" deydi. Lekin bu noto'g'ri. Avval reja kerak — qanday testlar, qancha vaqt, qancha odam, qaysi resurslar. Bu — Test Plan.

Test Plan — sizning loyihangizdagi QA strategiyangizning hujjati. Uni o'qiganda hamma tushunadi: "Bu loyihada QA qanday ishlaydi".

💡 Oddiy misol
Uy qurish
Quruvchi to'g'ridan-to'g'ri g'isht terib boshlaydimi? Yo'q! Avval chizma: necha xona, qancha pul, necha ish kuni, qaysi materiallar. Test Plan ham xuddi shunday — QA chizmasi. Reja bilan ish — chiroyli va aniq. Reja'siz — chalkash.
Asosiy atama

Test Plan — loyihada nimani, qanday va qachon testlash haqidagi rasmiy hujjat. Odatda Google Docs yoki Confluence'da yoziladi.

Oddiy qilib: "QA ning loyiha rejasi" — testor jamoasining ish strategiyasi.

Kim yozadi?

Test Plan komponentlari

Standart Test Plan 6 ta asosiy bo'limdan iborat. Har biri loyihaning bir tomonini yoritadi.

1
Scope (Qamrov)
"NIMANI test qilamiz?"
Loyihaning chegaralarini belgilash. Nima ichida, nima tashqarida.
Misol: TestShop loyihasi
✅ Test qilinadi (In Scope):
  • Bosh sahifa, Login, Register
  • Mahsulot katalogi, Savat
  • To'lov sahifasi
  • Mobil va Desktop versiyalar
❌ Test qilinmaydi (Out of Scope):
  • Admin panel (alohida loyiha)
  • 3-tomon to'lov shlyuzi (boshqa jamoa)
  • Email yuborish tizimi
2
Test Strategy
"QANDAY test qilamiz?"
Qaysi turdagi testlar, qaysi darajalar, qaysi yondashuvlar. 10-darsda ko'rgan turlar shu yerda qo'llaniladi.
Misol: TestShop strategiyasi
  • Functional Testing — Manual + Avtomatlash (Selenium)
  • UI Testing — Manual
  • Performance Testing — JMeter (Senior QA)
  • Security Testing — Asosiy darajada (SQL injection)
  • Compatibility — Chrome, Safari, Firefox + iOS, Android
3
Resources & Schedule
"KIM va QACHON?"
Jamoangiz, vaqt jadvali, kerakli asboblar.
Misol: TestShop resurslari
👥 Jamoa:
  • 1 QA Lead (Test Plan + Strategy)
  • 2 Manual QA (test bajarish)
  • 1 Automation QA (Selenium scripts)
📅 Jadval:
  • 1-hafta: Test Analysis va Design
  • 2-3 hafta: Test bajarish
  • 4-hafta: Regression + UAT
🛠 Asboblar:
  • Jira (bug tracking), TestRail (test management)
  • Selenium, Postman, BrowserStack
4
Risks & Mitigation
"NIMA xato bo'lishi mumkin?"
Mumkin bo'lgan muammolar va ularning yechimi. Risklar 2 turga bo'linadi.
🛒 Product Risk

Mahsulot bilan bog'liq

  • To'lov tizimi ishlamasligi
  • Sayt sekinligi
  • Foydalanuvchi ma'lumotlari yo'qolishi
📅 Project Risk

Loyiha bilan bog'liq

  • Vaqt yetmasligi
  • QA jamoa kasallikka chiqishi
  • Talablar o'zgarishi
Mitigation (yechim) misoli
Risk: "QA jamoasi 1 hafta kech qoladi"
Mitigation: "Eng muhim funksiyalarni avval test qilamiz (Risk-based testing). Critical bo'lmaganlarni keyingi sprintga o'tkazamiz."
5
Entry / Exit Criteria
"QACHON BOSHLAYMIZ va TUGATAMIZ?"
Test boshlash va tugatish uchun aniq shartlar.
Entry Criteria — boshlash uchun
  • ✅ Build dasturchidan kelgan va deploy qilingan
  • ✅ Test environment tayyor
  • ✅ Test data tayyorlangan
  • ✅ Test case'lar yozilgan va review qilingan
Exit Criteria — tugatish uchun
  • ✅ Barcha P1/P2 buglar tuzatilgan
  • ✅ 95% test case Pass holatida
  • ✅ Regression Testing bajarilgan
  • ✅ Test Summary Report tayyor
6
Deliverables
"NIMA TOPSHIRAMIZ?"
Test natijasida QA jamoa beradigan hujjatlar va materiallar.
Misol: TestShop deliverables
  • 📄 Test Plan (bu hujjat)
  • 📋 Test Cases — TestRail'da 200+ ta
  • 🐛 Bug Reports — Jira'da
  • 📊 Test Summary Report — natijalar va statistika
  • 🤖 Automation scripts — Git repository'da

Intervyu savollari

Test Plan — Middle QA intervyusining asosiy mavzusi. 7 ta professional javob.

💡 Har savol ustiga bosing — professional javobni ko'rasiz.

Q1 Test Plan nima va nega kerak? +

Javob: "Test Plan — loyihada nimani, qanday va qachon testlash haqidagi rasmiy hujjat. QA strategiyasining asosi."

Nega kerak?

  • Reja: Jamoaga aniq yo'l beradi
  • Resurs: Vaqt va odamni rejalashtirish
  • Risk: Muammolarni oldindan ko'rish
  • Hujjat: Stakeholder'lar uchun

Pro javob: "Test Plan — bu QA jamoasining biznes plani. U bo'lmasa — chalkashlik, vaqt yo'qotish, sifat past bo'ladi."

Q2 Test Plan komponentlari qaysilar? +

Javob: "Standart Test Plan 6 ta asosiy bo'limdan iborat:"

  • 1. Scope — nima testlanadi (in/out of scope)
  • 2. Test Strategy — qaysi turlardagi testlar
  • 3. Resources & Schedule — kim, qachon, qaysi asboblar
  • 4. Risks & Mitigation — mumkin bo'lgan muammolar
  • 5. Entry/Exit Criteria — boshlanish va tugatish shartlari
  • 6. Deliverables — yakunda nima topshirasiz

Standart: IEEE 829 — bu xalqaro standart. Lekin har kompaniya o'zicha o'zgartiradi.

Q3 Entry va Exit Criteria nima farqi? +

Javob:

  • Entry Criteria — test boshlash uchun shartlar. "Qachon biz testni boshlashimiz mumkin?"
  • Exit Criteria — test tugatish uchun shartlar. "Qachon biz testni tugatamiz?"

Misollar:

  • Entry: Build deploy qilingan, test environment tayyor, test cases yozilgan
  • Exit: 95% test Pass, P1/P2 bug yo'q, Regression bajarilgan

Muhim: Har ikkisi o'lchanadigan bo'lishi kerak. "Yaxshi sifat" emas — "95% test Pass".

Q4 Product Risk va Project Risk farqi? +

Javob:

  • Product Risk — mahsulot bilan bog'liq risklar. Foydalanuvchiga ta'sir qiluvchi.
  • Project Risk — loyiha bilan bog'liq risklar. Jadvalga ta'sir qiluvchi.

Product Risk misollari:

  • To'lov tizimi ishlamasligi
  • Foydalanuvchi ma'lumotlari yo'qolishi
  • Sayt sekinligi

Project Risk misollari:

  • Vaqt yetmasligi
  • Jamoa kasalligi
  • Talablar o'zgarishi

QA roli: "Men Product Risk'larga e'tibor beraman. Project Risk asosan PM ishi, lekin QA Lead ham qatnashadi."

Q5 Risk-based testing nima? +

Javob: "Risk-based testing — eng xavfli funksiyalarni avval testlash strategiyasi."

Asosiy g'oya: Vaqt cheklangan — barcha narsani test qilib bo'lmaydi. Demak — eng muhim joylarni avval.

Qanday aniqlanadi?

  • Foydalanuvchi uchun muhimligi (to'lov vs profil)
  • Bug ehtimolligi (yangi kod vs eski)
  • Bug ta'siri (critical vs minor)

Misol: "Click loyihasida pul o'tkazish — yuqori risk. Profil rasmini o'zgartirish — past risk. Avval pul o'tkazishni test qilamiz."

Q6 Test Plan'ni kim yozadi? +

Javob: "Bu kompaniya hajmi va loyiha murakkabligiga bog'liq:"

  • Test Lead / QA Lead — katta loyihalarda asosiy mualif
  • Senior QA — kichik loyihalarda
  • Middle QA — kichik feature'lar uchun mini Test Plan
  • Junior QA — odatda yozmaydi, lekin o'qishi shart

Junior'lar uchun maslahat: "Men Test Plan'ni kuzatib boraman, savol beraman. Bu mening Middle'ga o'tish yo'lim."

Pro javob: "Birinchi loyihamda men kichik feature uchun mini Test Plan yozdim — QA Lead yordam berdi. Bu juda foydali tajriba edi."

Q7 Yangi loyihaga kelganda birinchi nima qilasiz? +

Javob: "Junior darajada javob: 'Test Case yozaman'. Lekin to'g'ri javob — boshqacha:"

  • 1. Test Plan'ni so'rayman va o'qib chiqaman
  • 2. Loyiha hujjatlari (PRD, BRD)'ni o'rganaman
  • 3. 3 Amigos meeting'larda qatnashaman
  • 4. Talablardagi noaniq joylarni aniqlash uchun savol beraman
  • 5. Test Strategy ga muvofiq Test Cases yozishni boshlayman

Muhim: "Reja'siz ish — chalkashlik. Reja bilan ish — professional natija."
