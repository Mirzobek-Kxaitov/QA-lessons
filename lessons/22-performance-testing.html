Nima uchun Performance Testing?

Siz testlarda hammasi yaxshi deb bildingiz. Lekin Black Friday kuni 10,000 odam saytga kirdi va sayt to'xtab qoldi. Kim aybdor?

Functional testing — sayt to'g'ri ishlayaptimi deb tekshiradi. Performance testing esa — sayt qanchalik tez va qancha yuklamaga chidaydi deb tekshiradi.

💡 Ko'prik misoli
Functional vs Performance
Functional testing: ko'prik to'g'ri qurilganmi? (1 mashina o'tadi). Performance testing: ko'prik 1000 ta mashinani ko'tara oladimi? Ikkalasi ham kerak!

Real misollar — Performance muammolari

  • Uzum Black Friday — ko'p foydalanuvchi kirdi, sayt sekinlashdi
  • Bank ilovasi — ish kuni boshida hamma kirmoqchi, server ishlamaydi
  • Online imtihon — bir vaqtda 5000 talaba kirdi, tizim to'xtadi
  • Onlayn do'kon — tezkor aksiya e'lon qilindi, sayt "tushib qoldi"
QA uchun ahamiyati

Performance muammolari release dan oldin topilishi kerak. Release dan keyin topilsa — biznes zarar ko'radi, mijozlar ketadi. Bu — QA ning muhim vazifasi.

TestShop uchun misol
  • Load: Kuniga 500 ta xaridor → kutilgan yuklama, normal ishlashi kerak
  • Stress: 5000 tagacha oshirib, sayt qachon javob bermay qolishini topamiz
  • Spike: "Chegirma 1 soat" e'loni — to'satdan 2000 kishi kiradi
  • Soak: 24 soat 200 foydalanuvchi — memory yoki server muammolari bormi?

Asosiy metrikalar

Performance test natijalarini o'qish uchun 4 ta metrikani bilish kifoya.

Response Time
2s
So'rovdan javobgacha vaqt. 2 sek yaxshi, 5 sek yomon, 10+ sek — foydalanuvchi ketadi.
Throughput
120
Sekundiga bajarilgan so'rovlar soni (req/sec). Ko'p = yaxshi.
Error Rate
1%
Xato qaytargan so'rovlar foizi. 1% dan kam — yaxshi. 5%+ — muammo.
Concurrent Users
500
Bir vaqtdagi foydalanuvchilar. Sayt qancha odamga bir vaqtda xizmat qilishi kerak?

p95 va p99 — nima bu?

Percentile tushunchasi

p95 = 1.8 sek degani: 100 ta so'rovdan 95 tasi 1.8 sekunddan tez javob berdi. Faqat 5 tasi sekinroq.

p99 = 3.2 sek degani: 99 tasi 3.2 sekunddan tez. Faqat 1 tasi sekinroq.

O'rtacha vaqt aldaydi — 1 ta juda sekin so'rov o'rtachani ko'tarmaydi. p95/p99 haqiqiy holatni ko'rsatadi.

Qabul qilish mezonlari (typik)

Response Time p95 < 2 sek · Error Rate < 1% · Throughput > 100 req/sec · Availability > 99.9%

Apache JMeter

JMeter — performance testing uchun eng mashhur bepul asbob. Apache Foundation tomonidan yaratilgan. 1998 yildan buyon.

JMeter nima?

Apache JMeter — Java'da yozilgan, bepul, ochiq kodli performance testing asbob. GUI va CLI rejimida ishlaydi.

Oddiy qilib: JMeter — minglab virtual foydalanuvchilarni yaratib, serverga so'rov yuboradigan dastur. Server qanchalik chidashini tekshiradi.

JMeter asosiy tushunchalari

  • Test Plan — testning umumiy rejasi (ildiz element)
  • Thread Group — virtual foydalanuvchilar guruhi. 100 thread = 100 foydalanuvchi
  • HTTP Sampler — bitta so'rov (GET /products, POST /login)
  • Listener — natijalarni ko'rsatuvchi grafik/jadval
  • Timer — so'rovlar orasidagi kutish vaqti
  • Assertion — javob to'g'riligini tekshirish

JMeter'da test yaratish — 6 qadam

1

JMeter'ni yuklab olish

jmeter.apache.org — Binary ZIP yuklab, Java kerak (JDK 8+). Mac'da: brew install jmeter

2

Test Plan yaratish

JMeter'ni oching → File → New. Test Plan nomini "TestShop Load Test" qiling.

3

Thread Group qo'shish

Test Plan ustiga o'ng klik → Add → Threads → Thread Group. Number of Threads: 100, Ramp-Up: 30, Loop Count: 5.

4

HTTP Request qo'shish

Thread Group ustiga o'ng klik → Add → Sampler → HTTP Request. Server: testshop.uz, Path: /api/products, Method: GET.

5

Listener (natija ko'rish) qo'shish

Thread Group → Add → Listener → Summary Report va View Results Tree. Natijalar shu yerda ko'rinadi.

6

Testni ishga tushirish

▶ Start tugmasi → JMeter 100 ta virtual foydalanuvchi bilan so'rov yuborishni boshlaydi. Summary Report'da natijalar ko'rinadi.

Intervyuga tayyorlov

Intervyu savollari

💡 Har savol ustiga bosing — professional javobni ko'rasiz.

Q1Performance testing nima va nima uchun kerak?+

Javob: "Performance testing — tizimning tezligi, barqarorligi va yuklamaga chidamliligini tekshirish."

Nima uchun kerak:

  • Functional test "ishlayaptimi?" deydi. Performance "qanchalik tez va qancha yuklamada?" deydi
  • Release'dan oldin bottleneck'larni topish
  • Server kapasitesini bilish (qancha foydalanuvchiga xizmat qila oladi?)

Pro javob: "Black Friday, yangi yil aksiyasi kabi yuqori yuklamalarda sayt buzilmasligi uchun oldindan test qilinadi."

Q2Load va Stress testing farqi?+
  • Load Testing — kutilgan (normal) yuklamada ishlash. "500 ta odimiy foydalanuvchi saytga kirganda hammasi yaxshimi?"
  • Stress Testing — chegarani topish. Yuklamani oshirib, tizim qachon "sindirini" aniqlaymiz
  • Spike Testing — to'satdan ko'p foydalanuvchi. "1 daqiqada 10x ko'p so'rov kelsa nima bo'ladi?"
  • Soak Testing — uzoq vaqt davomida. "24 soat ishlasa memory leak bo'ladimi?"
Q3JMeter'da Thread Group nima?+

Javob: "Thread Group — JMeter'da virtual foydalanuvchilar guruhi. Har thread — bitta foydalanuvchi."

Asosiy sozlamalar:

  • Number of Threads — nechta virtual foydalanuvchi
  • Ramp-Up Period — foydalanuvchilar necha sekundda qo'shilsin (asta-sekin yoki birdaniga)
  • Loop Count — har foydalanuvchi necha marta takrorlaydi

Misol: 100 thread, 30 sek ramp-up = har 0.3 sekundda 1 foydalanuvchi qo'shiladi. Real hayotga yaqin!

Q4p95, p99 nima degani?+

Javob: "p95 — percentile. p95 = 1.8 sek degani: 100 ta so'rovdan 95 tasi 1.8 sekunddan tez bajarildi."

Nima uchun o'rtacha yetarli emas:

  • O'rtacha (Avg) 300ms — yaxshi ko'rinadi
  • Lekin 5 ta so'rov 10 sekunddан ko'p — foydalanuvchi ketadi
  • p95/p99 bu 5%/1% "yomon" holatni ko'rsatadi

Pro javob: "Men doim p95 va p99 ko'raman. O'rtacha aldaydi."

Q5JMeter natijalarini qanday tahlil qilasiz?+

Javob: "Summary Report'dan 4 ta narsaga e'tibor beraman:"

  • Avg (ms) — o'rtacha response time. < 2000ms yaxshi
  • Error% — xato foizi. < 1% yaxshi, 5%+ — muammo
  • Throughput — sekundiga so'rovlar. Ko'p = server kuchli
  • Max — eng sekin so'rov. Juda yuqori bo'lsa — timeout muammosi

Qabul qilish mezonlari bilan solishtiramiz: "p95 < 2 sek va Error Rate < 1% — test o'tdi deb hisoblanadi."

Q6Bottleneck nima va qanday topasiz?+

Bottleneck — tizimning eng sekin qismi. Hamma yuklama shu yerda to'planadi.

Topish usullari:

  • JMeter'da qaysi endpoint eng sekin — /checkout 3820ms, /products 342ms → checkout bottleneck
  • Server monitoring (CPU, RAM, disk I/O) — qaysi resurс 100% ga yetadi?
  • Database query'lar — sekin query'lar response time oshiradi

Dasturchi bilan: "Bottleneck topildi → dasturchiga report → optimizatsiya → qayta test."

Q7Performance testing tajribangiz bormi?+

Pro javob:

"Men JMeter bilan load test o'tkazganman. TestShop API'iga 100 ta virtual foydalanuvchi bilan test qildim:"

  • Asbob: Apache JMeter
  • Test turi: Load Test — 100 foydalanuvchi, 30 sek ramp-up
  • Target: GET /products, POST /login endpointlari
  • Natija: /products — 342ms ✅, /login — 1240ms ⚠️
  • Xulosa: Login sekin, optimizatsiya tavsiyasini dasturchiga yubordim

Bu — Junior darajasida zo'r tajriba. Amalda bajarganman.
